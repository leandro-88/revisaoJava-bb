0.0 Fundamentos do Java
O Ecossistema da Linguagem
Máquina Virtual, Compilação e Bytecodes
📦 A Matrioska do Java (As Três Siglas):
• JVM (Java Virtual Machine): É a máquina virtual. É ela quem lê e executa o código. Pegadinha: A JVM NÃO é multiplataforma! Você precisa baixar uma JVM específica para Windows, outra para Linux, etc.
• JRE (Java Runtime Environment): É o ambiente de execução. Contém a JVM + Bibliotecas da API. É o que o usuário final instala para rodar o programa.
• JDK (Java Development Kit): É o kit do desenvolvedor. Contém o JRE + Ferramentas de criação (como o compilador javac e o javadoc).
⚙️ O Fluxo: .java ➔ .class
O Java não compila para código de máquina (como .exe) direto. Ele gera um código intermediário.
MeuCodigo.java (Código Fonte)
⬇️ Compilador (javac)
MeuCodigo.class (Bytecode)
⬇️ Interpretador (java)
JVM (Executa na RAM)
🚨 Radar Cesgranrio: WORA e JIT Compiler
• WORA (Write Once, Run Anywhere): "Escreva uma vez, rode em qualquer lugar". Como o compilador gera Bytecodes independentes, o mesmo arquivo .class roda no Windows, Mac e Linux, desde que tenham uma JVM [1].
• JIT (Just-In-Time) Compiler: A banca diz que "o Java é lento porque é 100% interpretado". Errado! A JVM usa o JIT para identificar Hot Spots (trechos de código muito usados) e traduzi-los dinamicamente para código de máquina nativo em tempo real, atingindo alta performance [1, 3].
Java 10/11 Sintaxe
var — Inferência de Tipo
O Compilador "Adivinha"
O QUE FAZ: O compilador deduz o tipo pela inicialização.
    var nome = "João"; // String
    var idade = 25; // int
    var lista = List.of(1, 2, 3); // List<Integer>
✅ PODE USAR var:
• Variáveis locais (dentro de métodos)
• Loops: for (var item : lista)
• (Java 11) Parâmetros de Lambda: (@NonNull var x) -> x + 1
❌ NÃO PODE USAR var:
• Atributos de classe (Variável de instância) ❌
• Parâmetros de métodos (exceto Lambda) ❌
• Sem inicialização: var x; ❌
• Com null: var x = null; ❌
• Múltiplas na mesma linha: var a=1, b=2; ❌
⚠️ PEGADINHAS DE PROVA:
1. Tipagem Estática: O tipo não muda após inicializar.
var x = 1; x = "A"; ❌ (Erro: int não vira String)
2. Diamante Vazio (<>): O compilador fica "cego" e assume Object.
var lista = new ArrayList<>(); ⚠️ (Infere ArrayList<Object>)
3. Foge do Polimorfismo: Assume a classe exata, não a interface.
var mapa = new HashMap<K,V>(); ⚠️ (Fica engessado como HashMap)
4. Arrays sem 'new': Não infere listas com chaves diretas.
var vet = {1, 2}; ❌ (Correto: var vet = new int[]{1, 2};)
5. Palavra Reservada: "var" pode ser nome de variável.
var var = 10; ✅ (Compila sem erros!)

Java 9+ Interfaces
Métodos Private em Interface
Reutilização Interna de Código
CRONOLOGIA:
Java 8: default e static
Java 9+: private (novo!)
interface MinhaAPI {
    default void fazer() {
        helper(); // Usa método privado
    }
    private void helper() { } // ✅ Java 9+
}
🚨 REGRAS DE COMPILAÇÃO:
• Corpo obrigatório: Método private não pode ser abstrato (sem chaves). ❌
• Regra do Static: Um método static da interface SÓ pode chamar helpers que sejam private static. Ele não enxerga o private comum! ⚠️
• Não é Herdado: A classe que implementa a interface NÃO consegue acessar esse método. 🚫
⚠️ PEGADINHA CESGRANRIO:
A banca colocará uma classe implementando a interface e tentará chamar obj.helper() no método main, ou tentará usar @Override no método private.

É Erro de Compilação! O método private serve estritamente para encapsular e compartilhar lógica entre os métodos default da própria interface, escondendo isso do implementador.
Java 8+ Lambdas
Expressões Lambda
O que o Compilador Rejeita?
🚨 ERROS CLÁSSICOS DE COMPILAÇÃO:
• Regra do Return: return exige chaves { } e vice-versa.
x -> return x; ❌ (Faltou chaves)
• Tipos exigem parênteses: Mesmo com apenas 1 parâmetro.
String s -> s; ❌ (Correto: (String s) -> s)
• Effectively Final: O lambda só lê variáveis de fora se elas NUNCA forem alteradas depois.
int n = 1; var f = () -> n; n = 2; ❌ (Erro na linha do lambda)
• Conflito de Escopo (Shadowing): O parâmetro não pode ter o mesmo nome de uma variável local já existente.
int x = 5; lista.forEach(x -> print(x)); ❌ (A variável 'x' já existe)
💡 Dica Cesgranrio: Em testes de mesa, quando a questão perguntar "Qual a saída do código?", antes de calcular a lógica matemática, varra o código em busca de conflitos de nome, retornos sem chaves ou variáveis não-finais sendo alteradas. Muitas vezes a resposta correta é simplesmente "Erro de Compilação".

Java 8+ Pipeline de Dados
Stream API
📊 O Pipeline "Preguiçoso"
REGRA DE OURO: Um Stream não armazena dados e SÓ é executado quando você chama uma Operação Terminal!
⚙️ INTERMEDIÁRIAS (Lazy / Preguiçosas) – Avaliam elemento a elemento, mas não processam até o Terminal
SEM ESTADO (Não precisa “lembrar” dos elementos):
  • .filter(n -> n > 5)  // Deixa passar apenas elementos que atendem à condição
  • .map(String::toUpperCase)  // Transforma cada elemento da entrada
  • .flatMap(lista -> lista.stream())  // Achata estruturas (ex: List> → List)
COM ESTADO (Precisa guardar elementos):
  • .distinct()  // Remove duplicatas (usa equals), precisa “lembrar” quem já passou
  • .sorted()  // Ordena elementos, guarda tudo até ordenar
  • .limit(3)  // Corta o fluxo após N elementos passarem
⚡ TERMINAIS (Eager / Gulosas) – Fecham o fluxo e produzem resultado
  • .collect(Collectors.toList())  // Materializa o resultado final em uma coleção (List, Set, Map)
  • .reduce(0, (a, b) -> a + b)  // Agrega elementos em um único valor (ex: soma total)
  • .count() / .forEach(System.out::print)  // Conta elementos ou executa ação final em cada elemento
🚨 RADAR CESGRANRIO:
Se um Stream tiver .map() seguido de .filter(), mas não houver operação Terminal (.collect(), .count(), .forEach()...), nada será processado! O fluxo é **preguiçoso** e só executa quando um Terminal é chamado.
💡 Dica Extra: Pense no Stream como uma esteira de fábrica: as operações intermediárias apenas preparam a linha, mas o produto só sai quando alguém pega no final (terminal).

Java 8+ Menos Verborragia
Method Reference ( :: )
Entenda quem chama e quem é parâmetro
COMO LER NA PROVA: O :: é puro "açúcar sintático". O elemento que está passando pelo tubo do Stream vai ser "injetado" nessa chamada, seja como parâmetro ou como o próprio dono da ação.
1. Instância Específica (O objeto já existe):
stream.forEach(System.out::println);
// O elemento 'x' do fluxo entra como parâmetro.
// É igual a: stream.forEach(x -> System.out.println(x));
2. Instância da Própria Classe (O elemento chama a ação):
stream.map(String::toUpperCase);
// O próprio elemento 's' do fluxo é quem CHAMA o método.
// É igual a: stream.map(s -> s.toUpperCase());
3. Método Estático (Utilitários da Classe):
stream.map(Math::sqrt);
// A classe não é instanciada. O elemento entra direto como argumento.
// É igual a: stream.map(n -> Math.sqrt(n));
4. Referência de Construtor (Fabricar objetos):
stream.collect(Collectors.toCollection(ArrayList::new));
// Invoca o construtor vazio "new ArrayList()" para guardar os dados.
// É igual a: () -> new ArrayList<>()
🚨 RADAR CESGRANRIO:
Se o método do Java precisa receber outros parâmetros extras além do próprio elemento do stream, você É OBRIGADO a usar a sintaxe Lambda normal! [1]
s -> s.substring(0, 5) NÃO pode virar String::substring, porque os números 0 e 5 não têm onde ficar na sintaxe reduzida. ❌
Java 8+ Anti-NullPointer
A Classe Optional
Uma "caixa" que pode ou não conter algo
O CONCEITO: Em vez de retornar null (que causa erros), os métodos agora retornam um objeto Optional. Você é obrigado a "abrir a caixa" do jeito certo para pegar o valor.
📦 CRIANDO A CAIXA (Atenção aqui!):
Optional.of(valor);
// ❌ PEGADINHA: Se 'valor' for null, lança NullPointerException na hora!

Optional.ofNullable(valor);
// ✅ SEGURO: Se for null, não dá erro. Ele apenas cria uma "caixa vazia" (Optional.empty).
🛠️ ABRINDO A CAIXA COM SEGURANÇA:
opt.orElse("Valor Padrão");
// Se tiver algo dentro, retorna o conteúdo. Se a caixa estiver vazia, retorna a String "Valor Padrão".

opt.orElseThrow();
// Tenta extrair. Se a caixa estiver vazia, "capota" lançando uma exceção imediatamente.

opt.ifPresent(System.out::println);
// Se (E SOMENTE SE) tiver valor, executa o Consumer (imprime na tela). Se vazio, não faz nada.
🚨 RADAR CESGRANRIO (A Armadilha do .get):
A banca vai colocar um código que cria um Optional vazio e chama o método de extração direta: opt.get().
Resultado: Como o programador não checou se estava vazio antes (com isPresent()), o Java lança a exceção NoSuchElementException. ❌
Java — Tratamento de Exceções
A Árvore de Exceções
O que o Compilador te obriga a tratar?
Throwable (A classe Mãe de todas)

  ├── Error (Ex: OutOfMemoryError)
  │      ↳ 🚫 NÃO SE CAPTURA! Erro grave na JVM.

  └── Exception (Exceções CHECKED. Ex: IOException)
         ↳ ⚠️ OBRIGATÓRIO: Exige try-catch ou throws.

         └── RuntimeException (Ex: NullPointer)
                ↳ ✅ UNCHECKED: Erros de lógica. Tratar é opcional.
⚖️ COMO LIDAR COM AS "CHECKED" (Obrigatórias):
Se o código dentro de um método gera uma exceção Checked, você tem 2 opções:
1. Assumir o B.O: Colocar dentro de um try-catch e resolver ali mesmo.
2. Passar a "Batata Quente": Colocar throws NomeDaExcecao na assinatura do método (Ex: void ler() throws IOException). Fazendo isso, você não precisa usar try-catch dentro do método, mas quem chamar este método será obrigado a tratar!
🚨 PEGADINHAS CLÁSSICAS CESGRANRIO:
  • O Mito do Finally: Mesmo que haja um return ou um erro dentro do try, o bloco finally SEMPRE será executado para limpar os recursos (a única exceção é se houver um System.exit(0)).

  • Try-with-resources (Java 7+): Ao abrir um recurso no parênteses: try (Connection c = ...), o Java dá o c.close() sozinho no final! Isso dispensa o uso do bloco finally!
Java — Exceções Avançadas
Try-with-resources & Hierarquia
As pegadinhas finais de compilação
🪄 TRY-WITH-RESOURCES (O Fim do Finally): Abre conexões no try ( ... ). O Java chama o .close() sozinho no final. Regra: O objeto TEM que implementar a interface AutoCloseable. [1]
🗂️ A ORDEM DOS CATCHS (A Hierarquia):
try {
    codigoPerigoso();
} catch (FileNotFoundException e) {
    // ✅ CORRETO: As exceções mais ESPECÍFICAS (Filhas) vêm PRIMEIRO!
} catch (IOException e) {
    // ✅ CORRETO: As mais GENÉRICAS (Pais) vêm DEPOIS para "pegar o resto".
}
❌ O ERRO CLÁSSICO: Se você inverter a ordem e colocar o Pai primeiro, as filhas nunca serão alcançadas (Unreachable code). O código NÃO COMPILA! [2]
🚨 HERANÇA VS EXCEÇÕES (O Filho não pode ser rebelde):
Quando uma classe Filha sobrescreve (@Override) um método da classe Pai, existem regras estritas de compilação para o throws:

1. Não pode inventar problemas novos: O Filho NÃO PODE lançar uma exceção Checked nova ou mais genérica (Mãe) que a declarada pelo Pai. [3]
2. O que o Filho PODE fazer: Lançar a mesma exceção, não lançar nada, ou lançar exceções mais específicas (Subclasses). [3]

Exemplo de Prova: Se o método do Pai declara throws IOException, o método do Filho PODE declarar throws FileNotFoundException (que é filha). Mas o Filho NÃO PODE declarar throws Exception (que é mãe). ❌
Java — A Maior Pegadinha
== vs .equals()
A Armadilha da Igualdade
A REGRA DE OURO:
• O == compara a REFERÊNCIA (Endereço na memória. É o exato mesmo objeto?)
• O .equals() compara o CONTEÚDO (As letras ou o valor lá dentro são iguais?)
🧠 TESTE DE MESA (O Paradoxo da String):
String s1 = "java";
String s2 = "java";
// O Java otimiza literais! s1 e s2 apontam pro mesmo objeto no "Pool de Strings".

String s3 = new String("java");
// O 'new' FORÇA a criação de um NOVO objeto em outro endereço de memória.

s1 == s2;       // ✅ TRUE (Apontam para a mesma memória)
s1 == s3;       // ❌ FALSE (Memórias diferentes por causa do 'new')
s1.equals(s3); // ✅ TRUE (Verifica as letras, o conteúdo é igual)
🚨 RADAR CESGRANRIO (Classes Wrapper):
As bancas amam fazer isso com números também. Se a prova colocar:
Integer a = new Integer(10);
Integer b = new Integer(10);
E depois testar a == b, o resultado é FALSE, pois cada new gera uma referência diferente na memória! Para objetos (String, Integer, Long), desconfie do == e SEMPRE USE O .equals()!
Java — Fundamentos OOP
Sobrecarga vs Sobrescrita
Nomes parecidos, regras de compilação diferentes
📦 SOBRECARGA (Overload) - Polimorfismo Estático:
// Mesma classe. Mesmo nome de método, mas PARÂMETROS DIFERENTES.
// O compilador (em tempo de compilação) decide qual chamar.
void imprimir(String s) { }
void imprimir(int i) { } // ✅ OK! Assinaturas diferentes.
🧬 SOBRESCRITA (Override) - Polimorfismo Dinâmico:
// Herança (Pai/Filho). Exatamente a MESMA ASSINATURA (Nome e Parâmetros).
class Pai { public void fazer() { } }

class Filho extends Pai {
    @Override // Opcional, mas protege de virar Sobrecarga por acidente
    public void fazer() { } // ✅ Substitui o comportamento do Pai
}
🚨 REGRA LETAL DE COMPILAÇÃO (Visibilidade):
Na Sobrescrita, o método do Filho não pode ser "tímido". Ele NÃO PODE SER MAIS RESTRITIVO que o método do Pai [4]!

Exemplo de Prova: Se o método no Pai é public, no Filho ele é obrigado a ser public. Se o Filho tentar declarar como protected ou private, dá Erro de Compilação imediato [5, 6]!
Stream API — Transformações
map() vs flatMap()
Entenda o "Achatamento" de dados
A REGRA BÁSICA: O map transforma o valor, mas mantém a estrutura intacta. O flatMap destrói as subestruturas (listas dentro de listas) e derrama tudo em um "balde" só.
📦 MAP (1 entrada → 1 saída exata):
List<String> nomes = List.of("Ana", "Bia");

List<Integer> tam = nomes.stream()
    .map(String::length)
    // "Ana" vira 3. "Bia" vira 3. Cada item é substituído pelo seu tamanho.
    .collect(Collectors.toList());
// RESULTADO: [2]
🥞 FLATMAP (1 entrada → N saídas):
List<String> frases = List.of("Java 8", "Cai muito");

List<String> palavras = frases.stream()
    .flatMap(f -> Arrays.stream(f.split(" ")))
    // 1. split() quebra "Java 8" no array ["Java", "8"]
    // 2. Arrays.stream() transforma esse array num mini-fluxo
    // 3. flatMap pega os itens desse mini-fluxo e joga no fluxo principal!
    .collect(Collectors.toList());
// RESULTADO: ["Java", "8", "Cai", "muito"] (Tudo achatado numa lista só)
🚨 RADAR CESGRANRIO (Erro de Compilação):
O método flatMap() EXIGE que a função lambda retorne um Stream [3]! Se no teste de mesa a banca colocar .flatMap(f -> f.split(" ")) sem o Arrays.stream() em volta, marque a opção "Erro de compilação", pois ele não aceita receber um Array direto, apenas um Stream!

Onde todos erram no Teste de Mesa

🚨 REGRAS LETAIS DE COMPILAÇÃO:
1. O Filho não pode ser mais restrito:
Se o método no Pai é public, o Filho @Override não pode virar protected ou private.

2. O Paradoxo do Private Abstract:
private abstract void fazer(); // ❌ ERRO! Se é privado, o filho não vê. Se é abstrato, o filho TEM que ver para implementar. O Java proíbe essa combinação!

3. O Mito do Construtor Abstrato:
A banca dirá que classes abstratas não têm construtores porque não podem ser instanciadas. FALSO! Elas possuem construtores, que são chamados internamente pelo super() das classes filhas concretas!
1. Sintaxe Moderna e Java SE 11
O Novo Padrão Java
var, Imutabilidade e Factory Methods
🚨 List.of() e a Falsa Adição:
A criação de coleções com métodos como List.of(), Set.of() ou Map.of() gera estruturas estritamente imutáveis [1]. A banca frequentemente coloca um lista.add("C") logo em seguida; isso não insere o item, mas sim lança uma UnsupportedOperationException em tempo de execução [1].
🕵️ O Uso Restrito do 'var':
A inferência de tipo local (usando var) cai muito. O compilador proíbe o uso do var como atributo de classe, como parâmetro de método (exceto em lambdas), ou quando inicializado com null e sem valor [1, 2].
🧊 A Ilusão da String:
A classe String é imutável em Java [3]. Qualquer método que pareça modificá-la (como s.toUpperCase()) na verdade devolve uma nova instância [3]. Ele nunca altera a variável original a menos que haja uma reatribuição explícita no código (ex: s = s.toUpperCase()) [3].
2. OOP e Polimorfismo
A Base Estrutural
Contratos, Heranças e Limitações
🧬 Herança vs Interfaces:
A banca testa o vocabulário básico. Para uma classe herdar de outra usa-se extends, mas para implementar uma interface usa-se obrigatoriamente implements [4, 5]. Cuidado: o Java não suporta herança múltipla de classes (para evitar o problema do diamante), mas permite que uma classe implemente várias interfaces [6, 7].
🎭 O Falso Polimorfismo:
Polimorfismo é um tema central [8]. O Java suporta o polimorfismo dinâmico clássico [8], mas a banca tentará te induzir ao erro afirmando que Java suporta "sobrecarga de operadores" ou "passagem de parâmetros de tipos primitivos por referência", recursos que a linguagem NÃO possui [6, 9].
🔒 Variáveis em Interfaces:
O compilador considera toda variável declarada dentro de uma interface implicitamente como public static final (constantes) [10, 11]. Se o código do teste de mesa tentar incrementar (MAX++) ou alterar essa variável, ocorrerá um Erro de Compilação [10, 11].
3. Tratamento de Exceções
A Hierarquia do Caos
Controlando Falhas de Execução
⚖️ Checked vs Unchecked:
Saber diferenciar as exceções é fundamental: as Checked (como IOException) obrigam o compilador a forçar você a tratá-las com try-catch ou declará-las no throws [12, 13]. Já as Unchecked (filhas de RuntimeException, como NullPointerException) são erros de lógica e o tratamento é opcional [12, 13].
🧹 O Poder Supremo do "finally":
O try-catch-finally tem presença garantida nos testes de mesa [14]. A instrução finally permite a execução de um bloco de código de limpeza de recursos independentemente de o try ter sucesso ou de um erro ter sido capturado no catch [14, 15]. Se houver um return dentro do finally, ele anula o return do try!

O Coração da Ciência da Computação

A banca mistura Java com Estrutura de Dados pura em testes de mesa pesados [19, 20]. Saber a diferença de comportamento estrutural é vital [19, 20]:

🌐 API REST Corporativa (JAX-RS):
O padrão Jakarta EE utiliza anotações para mapear URIs diretamente para métodos. O @Path define a URL, o @PathParam extrai variáveis da rota e o @Produces indica o formato de saída (ex: XML ou JSON).
@Path("/clientes")
public class ClienteResource {

    @GET
    @Path("/{id}")
    @Produces(MediaType.APPLICATION_JSON)
    public Cliente buscar(@PathParam("id") Long id) {
        // Lógica para buscar cliente e devolver como JSON
    }
}
🍃 API com Spring MVC:
No Spring, evitamos renderizar páginas HTML (JSPs) em APIs REST usando a anotação @ResponseBody. Ela diz ao framework que o objeto retornado pelo método deve ser serializado e inserido diretamente no corpo da resposta HTTP (geralmente com Status 200 OK).
@Controller
public class TarefaController {

    @ResponseBody
    @RequestMapping("finalizaTarefa")
    public void finaliza(Long id) {
        // A ausência de retorno + @ResponseBody gera um HTTP 200
    }
}
☁️ Serverless em Java (Aviso: Fonte Externa):
Em uma arquitetura Serverless (como AWS Lambda), você não configura um servidor Tomcat ou Jetty. Você escreve apenas uma função isolada que é ativada por um gatilho (evento). A nuvem sobe a máquina, roda a função e desliga em milissegundos.
// Exemplo Externo: AWS Lambda em Java
public class MinhaFuncao implements RequestHandler<Request, Response> {

    @Override
    public Response handleRequest(Request req, Context context) {
        // O servidor não existe o tempo todo. Ele só "acorda" aqui!
        return new Response("Executado na Nuvem de forma Serverless!");
    }
}

System.out.println(res.getBody()); // Imprime o corpo da resposta
